A Tutorial on Understanding and Managing “Documentation” Across Multiple Teams

This tutorial provides an overview of how “documentation” can be interpreted differently within an organization and offers strategies for managing documentation efforts across varied teams—especially when there’s an auditing or compliance requirement in play.

1. Background: Why Documentation Matters for Auditing

  1. Accounting Perspective

    • When an IT department builds software or other technology “in-house,” that work may be treated as an asset on the company’s books.
    • Accounting needs evidence that the intangible asset genuinely exists and that resources were appropriately allocated to create it.

  2. Auditing Perspective

    • Auditors verify that the work claimed to have been done (which may be capitalized) has indeed been completed.
    • Tools like Azure DevOps Boards store tasks, user stories, sprints, and other records showing how and when work was completed.
    • This “paper trail” is critical: it provides proof of the tasks done, the code changes, and the testing completed—essentially confirming that the intangible asset truly exists.

2. Recognizing Different Types of Documentation

The term “documentation” can be confusing because it’s used in multiple ways for multiple audiences. Common documentation categories include:

  1. Technical/Developer Documentation

    • Written by developers for developers.
    • Explains code structure, design decisions, APIs, how modules integrate, etc.
    • Main goal: Make future maintenance and extensions easier.

  2. User Documentation

    • Created for end users or business stakeholders who need to operate or consume a product.
    • May include step-by-step instructions, FAQs, screenshots of workflows, etc.

  3. Process/Project Documentation (e.g., Azure DevOps Boards)

    • Documents how tasks, user stories, or product increments are planned and completed.
    • Useful for project managers, auditors, or anyone needing an overview of the work’s progress and sign-offs.

  4. Audit/Compliance Documentation

    • Specifically addresses regulatory and audit needs.
    • Focuses on proving that proper controls, processes, and sign-offs were followed.
    • Demonstrates compliance with internal or external standards.

Depending on the organization and the project, these types of documentation may overlap or even be combined. However, each typically serves a distinct audience or purpose.

3. Real-World Scenario: Four Different Teams, Four Different Documentation Needs

Imagine a department with four teams:

  1. Data Governance – Creates data policies, standards, and compliance guidelines.
  2. Data Analytics – Works on statistical or machine-learning models, advanced analytics, or data-driven insights.
  3. Data Engineering – Builds and maintains data pipelines, ingestion processes, ETL/ELT frameworks, and often handles data infrastructure.
  4. Business Intelligence (BI) – Creates dashboards, visualizations, and reports (e.g., in Power BI).

Each of these teams requires documentation, but the form and content of that documentation can vary:

  • Data Governance Documentation might be policy manuals, frameworks for handling data quality and compliance, or records of data usage and retention policies.
  • Data Analytics Documentation might describe methodologies for models, data sources used, code snippets for reproducible research, or Jupyter notebooks.
  • Data Engineering Documentation might detail pipeline architectures, code for ETL processes, infrastructure diagrams, or DevOps pipeline logs.
  • BI Documentation might include user guides for dashboards, instructions on how to refresh or maintain the reports, and the logic behind KPIs.

4. Challenges: Lack of Manager Involvement & Standardization

When you are tasked with creating a “documentation strategy” for the entire department, you can face issues such as:

  • Different Teams, Different Needs
    The diverse nature of the teams makes it tricky to create a single, one-size-fits-all approach.
  • No Managerial Sponsorship
    Without leadership support, it can be hard to mandate consistent documentation practices.
  • Unclear Ownership
    It’s often unclear who is accountable for ensuring that documentation is maintained and meets the required standards.

5. Practical Strategies to Move Forward

  1. Seek Management Buy-In

    • Present a concise summary of the benefits and risks:
      • Benefits: easier onboarding, reduced maintenance time, clearer audit trails.
      • Risks of doing nothing: unverified work, inconsistent processes, difficulty passing audits.
    • Even a short manager-approved plan can bolster cooperation across teams.

  2. Set Minimal, Common Requirements

    • Identify a few “universal” standards (e.g., a shared repository, basic versioning rules, naming conventions).
    • Don’t over-prescribe. Let each team add specialized sections or details they need.

  3. Use a Standard Template with Space for Customization

    • Include basic fields like Title, Owner, Revision Date, Purpose, and Dependencies.
    • Encourage each team to add sections relevant to their deliverables (e.g., policy vs. code docs vs. dashboards).

  4. Involve Key Representatives from Each Team

    • If direct managers aren’t available, aim to get input from senior contributors or team leads.
    • Ask them to champion consistent documentation practices within their team.

  5. Pilot and Iterate

    • Start with a single, smaller project or a single team.
    • Gather feedback on the process, fix pain points, then roll out a refined version department-wide.

  6. Clarify Auditing vs. Technical Documentation

    • Recognize that “documentation for audit” (tasks in Azure DevOps, compliance checklists) can be separate from “developer documentation” (code-level explanations, architecture diagrams).
    • Overlap may exist, but the audience and purpose differ.

6. Conclusion

“Documentation” is a broad term that means different things to different people. In an environment with diverse teams (Data Governance, Analytics, Engineering, BI), each producing different deliverables for various audiences (internal developers, auditors, end users), you need a flexible but standardized approach. The key is to:

  1. Obtain buy-in from leadership to validate documentation requirements.
  2. Create a minimal, shared structure that still allows each team to adapt as needed.
  3. Iterate and refine—documentation practices often evolve over time as the teams and their deliverables change.

This balanced approach helps ensure each team’s work is clearly captured for both day-to-day operational benefits (future maintenance, project handovers) and broader organizational needs (audit compliance, intangible asset verification).

2 A Tutorial on Overcoming the “Wicked Problem” of Documentation, Tapping Subject Matter Experts, and Avoiding a Low Bus Factor

Organizations often struggle to implement effective documentation strategies, especially when diverse teams and evolving requirements are involved. In this tutorial, we’ll explore three interrelated challenges:

  1. The “Wicked Problem” nature of creating a documentation solution.
  2. The dilemma with subject matter experts (SMEs) who already “know the answer.”
  3. The “bus factor,” which highlights the risk of single-person dependencies.

By understanding these concepts and employing practical strategies, you can improve how your organization captures and retains knowledge.

1. Understanding the Wicked Problem of Documentation

What is a Wicked Problem?
A wicked problem is one where the requirements are unclear or change over time, and there’s no single “right” solution. Documentation in a multi-team environment (e.g., Data Governance, Data Analytics, Data Engineering, and Business Intelligence) exemplifies a wicked problem because:

  • Multiple Stakeholders, Multiple Needs
    Each team has different deliverables and audiences for documentation.
  • Unclear or Evolving Constraints
    Organizational priorities, technology stacks, and personnel often shift, requiring documentation practices to adapt.
  • No Definitive Finish Line
    A single approach might work momentarily but must be revisited or iterated on as the organization grows.

Practical Tips for Tackling This Wicked Problem

  1. Start Small and Iterate
    • Implement a pilot documentation standard on a limited project. Gather feedback and refine before rolling out more broadly.
  2. Establish Shared Minimums
    • Define a core “template” or “checklist” for everyone to follow (e.g., file naming conventions, required fields like “owner” and “last updated”), but allow each team to add or tailor sections to their unique context.
  3. Seek Managerial or Leadership Buy-In
    • The complexity of a wicked problem usually demands leadership support. Show the risks of not documenting (e.g., bottlenecks, delayed onboarding, audit issues) and ask for a mandate that encourages team participation.

2. Subject Matter Experts: The Paradox of Knowledge Holders

The SME Dilemma
The person most qualified to document a system or process is often the person least motivated to do so. They already know the details by heart, so writing it all down can feel like an unnecessary extra step—particularly under tight deadlines.

Strategies to Capture SME Knowledge

  1. Pair Writing
    • Match the SME with a colleague who needs the knowledge. The SME explains verbally while the colleague documents in real time. This leverages the SME’s expertise without making them handle all the clerical work.
  2. Documentation Sprints
    • Allocate short, dedicated sessions for SMEs to share important information. Make sure someone else is capturing the details in a shared repository.
  3. Incentives and Cultural Norms
    • Encourage (or require) SMEs to document by tying it to performance reviews, career progression, or by highlighting how good documentation reduces long-term interruptions for everyone (including the SME).
  4. Ongoing Maintenance
    • Foster a “leave it better than you found it” mindset, where any time the SME (or anyone) modifies a piece of code or process, they add or update documentation accordingly.

3. The Bus Factor: Avoiding Single Points of Failure

What is the Bus Factor?
Also known as a “knowledge single-point-of-failure,” the bus factor quantifies how many people on a team can be “lost” (due to departure, leave, sudden unavailability) before a project is at risk. A bus factor of 1 means only one person understands a critical process—so if that person disappears, the project stalls.

Why a Low Bus Factor is Risky

  • Operational Delays
    Getting a newcomer up to speed can be slow and error-prone if no one else has the necessary knowledge.
  • Increased Pressure on Other Team Members
    Others must scramble to learn or maintain the abandoned system—often at the expense of their own tasks.
  • Potential for Critical Errors
    Without the original expert, new maintainers might introduce mistakes simply because they lack the big-picture understanding.

Strategies to Improve the Bus Factor

  1. Cross-Training
    • Rotate tasks or pair up team members so that knowledge is shared organically.
  2. Job Shadowing
    • Encourage occasional “shadow” days where one team member observes an SME’s routine tasks.
  3. Shared Repositories and Version Control
    • Keep code, diagrams, and process docs in a central, accessible place. The more transparent the workflow, the easier it is for a second person to learn.
  4. Regular Knowledge-Sharing Meetings
    • Hold short, recurring sessions where teammates explain a specific process or update. This raises overall team awareness and reduces single-person risk.

4. Bringing It All Together

1. Recognize the Complexity (Wicked Problem)
Documentation across different teams won’t be solved overnight. Understand that it’s an ongoing challenge requiring leadership commitment, iterative improvement, and flexible frameworks.

2. Encourage SMEs to Share
Adopt practical methods—pair writing, documentation sprints, cultural incentives—to capture crucial knowledge from the experts who are otherwise least inclined to write it down.

3. Reduce Single Points of Failure (Bus Factor)
Consciously distribute knowledge across multiple people, ensuring at least one backup per critical domain. Cross-training and collaborative documentation practices help increase your bus factor.

5. Conclusion

Documentation is both an art and a strategy, made more complex by differing team needs, SME bottlenecks, and the risk of losing critical knowledge if key people leave. By viewing the challenge as a wicked problem, employing structured methods to capture SME expertise, and boosting your bus factor, you create a more resilient, adaptive organization—one better able to respond to change and maintain continuity of operations.

Project Description: Documentation Centralization & Standardization

Background & Objective

The Corporate Data & Analytics department is tackling longstanding issues with how we create, store, and access documentation. Currently, critical information is scattered across various platforms (e.g., Azure DevOps, SharePoint, email, chat), and there is no consistent method to find or create new documentation. Our goal is to establish a centralized, standardized, and user-friendly system that enables easier knowledge sharing, ensures continuity, and supports our auditing requirements.

Proposed Solution

  1. Central Knowledge Base
    Store all departmental documentation in a structured format (e.g., Markdown) within Azure DevOps Repos.
  2. Consistent Templates
    Develop a small set of standardized templates for key documentation types (e.g., policies, processes, technical guides).
  3. User-Friendly Website
    Build a pipeline that converts Markdown files in the repo into a browsable website, allowing for simple navigation and search.
  4. AI-Enabled Workflows (Longer Term)
    • Speech-to-Text + Templates: Streamline document creation by taking audio input, converting it to text, and using an AI tool to format that text into standardized Markdown.
    • Retrieval-Augmented Generation (RAG): Provide an AI-based Q&A interface that retrieves relevant documents from the knowledge base to answer user queries.

Current State

  1. Initial Platform Setup
    • An Azure DevOps project has been created with a dedicated repository for Markdown files.
    • Basic knowledge of Markdown has been shared among team members.
  2. Templates in Development
    • A few core templates (e.g., “Process Documentation,” “Policy,” “Technical Write-up”) are being drafted and tested.
    • Each team (Business Intelligence, Data Analytics, Data Governance, and Data Engineering) is providing feedback to ensure these templates meet their needs.
  3. MVP Planning
    • The Minimum Viable Product (MVP) is scheduled for completion in about two weeks.
    • The MVP will include:
      • A small set of Markdown documents demonstrating the new format.
      • A pipeline to publish these documents to a basic website or static page.
      • Optionally, a simple AI prototype (RAG or speech-to-text workflow) using generic, non-sensitive data.
  4. Security & Compliance
    • Sensitive or proprietary information will not be included in the MVP.
    • The team is engaging cybersecurity to define future guidelines for handling confidential content.

Next Steps

  1. Finalize MVP Scope
    • Confirm exactly which templates and documents will be included.
    • Set up the pipeline for auto-publishing Markdown to a web interface.
  2. Conduct Template Trials
    • Gather real or representative content from each team to validate templates.
    • Incorporate feedback to refine the structure and formatting.
  3. Prototype AI Integration
    • Demonstrate how an AI tool could transform raw text (e.g., speech-to-text) into the standardized templates.
    • Explore a basic Q&A feature (RAG) using non-sensitive documentation samples.
  4. Plan for Rollout & Governance
    • Discuss how documentation ownership and periodic reviews will ensure accuracy over time.
    • Evaluate longer-term licensing and cost considerations for the full solution.

Contact/Questions

  • If you have questions about the MVP timeline, templates, or AI experiments, reach out to the project team.
  • For concerns about security or data sensitivity, please coordinate with our cybersecurity liaison.

Summary
This project aims to centralize and streamline how we handle departmental knowledge. By creating standardized templates, storing everything in a single repository, and eventually leveraging AI for faster document creation and retrieval, we’ll reduce duplicated effort, minimize knowledge gaps, and improve overall efficiency and compliance. The upcoming MVP demonstration will showcase our progress and serve as a foundation for continued development.
