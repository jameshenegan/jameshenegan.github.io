GitHub

Yes, it’s absolutely possible to use GitHub for documentation. In fact, there are two main ways to store documentation on GitHub:

  1. GitHub Wikis

    • Each GitHub repository can have its own wiki, which is essentially a mini documentation site.
    • The wiki is backed by a separate Git repository, so you can clone it, edit pages locally, and push changes.
    • You can write pages in Markdown, create links between pages, and even restrict access in private repositories (if you’re using GitHub Enterprise or a paid plan).
    • Pros: Easy to set up, integrates naturally with the project’s main repo, and supports version history.
    • Cons: Format and styling options are limited compared to dedicated documentation platforms, and navigation can feel less robust for very large documentation sets.

  2. GitHub Pages (Static Site Hosting)

    • You can use GitHub Pages to host a static site generated from your repo (for instance, using a static site generator like Jekyll, Docusaurus, or MkDocs).
    • This allows you to create a more polished, user-friendly documentation website, often with theming, a sidebar navigation, search, and other features.
    • Pros: Highly customizable, good for open-source or public-facing documentation, integrates well with Git-based workflows.
    • Cons: Requires some setup—learning or configuring a static site generator. Also, private repositories require a paid plan to use GitHub Pages privately (unless you’re on GitHub Enterprise).

Why Use GitHub for Documentation?

  • Version Control: Git-based workflows make it easy to track changes, revert to older versions, and enforce pull request reviews for edits.
  • Developer Familiarity: Many technical teams already use GitHub for code; documentation stored alongside code can streamline contributions.
  • Cost-Effectiveness: For public/open-source projects, GitHub Wikis and GitHub Pages are generally free to use.

Potential Drawbacks

  • Collaboration Model: Non-developers might find Git-based workflows less intuitive than a point-and-click wiki editor like Confluence or Notion.
  • Limited Rich Media / Structure: GitHub Wikis don’t always have the same advanced features (e.g., drag-and-drop formatting, inline commenting, dynamic page hierarchies) that dedicated documentation tools provide.
  • Access Control & Permissions: GitHub’s permission model might be more granular in Enterprise-level plans, but for smaller setups, you typically have to grant read/write to the whole repo.

In short: GitHub definitely supports documentation, either through its Wiki feature (simple, built-in wiki pages) or GitHub Pages (static site hosting for more polished docs). If your team is already comfortable with Git and primarily writes technical content, it can be a solid, cost-effective solution for version-controlled documentation. If you need a more user-friendly, non-developer-focused interface or robust wiki features, you might prefer specialized documentation platforms.

Azure DevOps

Yes, you can absolutely store Markdown files in an Azure DevOps repo and then render them into a documentation site. This approach is similar to using Git-based wikis (like GitHub Wikis or GitBook) but runs in the Azure DevOps ecosystem, which you might already be using for source control, CI/CD pipelines, and project management. Below are some things to consider:

1. Using Azure DevOps Wiki vs. Markdown in a Repo

  1. Azure DevOps Wiki

    • Azure DevOps provides a built-in wiki feature where each page is written in Markdown.
    • It’s easy to set up, and the wiki is automatically linked with your project.
    • You don’t have to manage a separate repository for the documentation—though it can be cloned and edited offline if needed.

  2. Markdown in a Repo

    • You can place your .md files directly in a repo (e.g., a dedicated “documentation” repo or your existing code repo).
    • This can be combined with pipelines to generate a static site (HTML) using a tool such as MkDocs, Docusaurus, or any other static site generator.
    • Once rendered, you can host the generated HTML pages in various ways:
      • Azure DevOps Artifacts (less common for hosting documentation)
      • Azure Storage + Static Website feature
      • Azure Web App (for more complex setups)

2. Setting Up a Static Site Generator

If you want to do the repo + pipeline + generated site approach:

  1. Pick a Static Site Generator (SSG)

    • MkDocs (simple Markdown-based docs, optional MkDocs Material theme for a professional look)
    • Docusaurus (popular for open-source or API docs, React-based)
    • Hugo, Jekyll, etc. (many options exist)

  2. Configure a Pipeline

    • In your Azure DevOps pipeline, install the SSG (e.g., pip install mkdocs or npm install docusaurus).
    • Run the build script to convert Markdown → HTML.
    • Output the generated site into a folder (usually site/ or build/).

  3. Deploy the HTML

    • Azure Static Web Apps: A straightforward option if you want a free-tier static site with automated deployment from Azure DevOps.
    • Azure Web App: You can zip up the site folder and deploy it as a web app.
    • Azure Storage: Enable static website hosting on a storage account, then upload the files.

3. Pros of Markdown + Azure DevOps Repo

  • Version Control & Familiar Workflow: All the benefits of Git-based versioning—pull requests, code reviews, branching, tagging releases—apply to docs.
  • Developer-Friendly: Markdown is lightweight and easy to use, especially for those accustomed to writing in plain text.
  • Full Customization: A static site generator allows you to tailor the look, navigation, and structure exactly how you want.
  • One-Stop Shop: Your code, issues, pipelines, and documentation can all live in Azure DevOps, simplifying access control and discoverability.

4. Cons / Considerations

  1. Learning Curve for Non-Tech Staff

    • Non-developers might find “clone repo → edit Markdown → push changes → pipeline build” more daunting than a point-and-click wiki editor.
    • You can mitigate this by using the Azure DevOps web interface to edit Markdown files, but the process may still be less intuitive than a dedicated documentation tool.

  2. Setup & Maintenance

    • You’ll have to configure a pipeline and maintain the static site generator. Small overhead, but it’s extra work.
    • If your documentation is large or you want advanced features (search, versioning, translations), you’ll need to customize your SSG accordingly.

  3. Lack of Real-Time Collaboration

    • Markdown in a repo doesn’t inherently provide real-time collaborative editing, unlike some wikis (Confluence, Notion). You can only see changes after they’re committed/pushed.

  4. Permission Granularity

    • Access to the repo is controlled at the project or branch level. If you have specialized permissions requirements (e.g., certain pages visible only to certain roles), you may need complex branching or multiple repos.

5. Putting It All Together

If your team is:

  • Comfortable with Git workflows (or at least can handle pull requests for doc changes).
  • Already using Azure DevOps for CI/CD and code management.
  • Interested in controlling the presentation and structure of your docs in a highly customizable way.

Then storing Markdown in an Azure DevOps repo, generating HTML via a pipeline, and hosting a static site is a very powerful method. You’ll get version control, a professional-looking site, and minimal licensing costs (since you’re leveraging your existing Azure DevOps subscription).

If, however, your organization has a large number of non-technical contributors or requires robust, user-friendly wiki editing, you might consider:

  • Azure DevOps Wiki for simpler, browser-based editing.
  • SharePoint with Word docs if your company is strictly Microsoft-based and prefers a more familiar interface.
  • A specialized tool like Confluence, Notion, or GitBook if you need advanced features (e.g., dynamic search, custom permission levels, or real-time collaboration).

Summary

Using Markdown in an Azure DevOps repo is a solid approach for a developer-centric documentation workflow, especially if you want to automate publishing a clean, static website. It provides robust version control and integrates seamlessly with your existing pipelines. The main trade-offs involve a slightly higher setup cost, less real-time collaboration, and a steeper learning curve for non-technical users.
