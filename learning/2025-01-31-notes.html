Second Meeting Notes

1. What to Document

  • Procedures, Workflows, and Processes

    • There was some confusion around these terms, prompting the question: Are they essentially the same, or do we differentiate them?
    • Concrete Example: In Data Governance, a “Data Quality Report” is a process worth documenting.

  • Regular Processes vs. One-Off Projects

    • Not all one-off projects need to be fully documented.
    • However, if a unique or complex issue arises in a one-off project, that learning should be documented to avoid “re-inventing the wheel” in the future.

2. Tools and Platforms

  • Obsidian

    • A personal knowledge base using Markdown files (exportable to PDF).
    • Question: Is Obsidian user-friendly enough for other departments with fewer technical skills?
    • Timeline Considerations: The priority this year is for the Corporate Data and Analytics teams to establish internal best practices. Introducing these tools to other departments may happen later.

  • SharePoint

    • Likely the primary hosting site for finalized documentation.
    • Potential Integration: Can Obsidian integrate seamlessly with SharePoint?

  • Other Departments’ Practices

    • Policy Services often uses Word, converting documents to PDF with screenshots.
    • Documentation formats can differ by team/department, but there should be a consistent “brand” or style.

3. Format and Consistency

  • Consistency vs. Flexibility

    • Every team’s documentation does not need an identical format, but branding should be consistent.
    • Each document should contain standard elements (“ingredients”) like Introduction, Summary, Purpose, Access Requirements, Code Snippets, etc.

  • Naming Conventions and Storage

    • Decide where documents are saved, how they are saved, and establish a clear naming convention to avoid confusion.

4. Past Documentation Experiences

  • Google Docs (Company A)

    • A main folder for documentation with sub-folders for big projects.
    • Files linked to each other, but it was challenging to maintain updated links and locate an “entry point” for each project.

  • Minimal or No Documentation (Company B)

    • Training was done in person.
    • Privacy issues prevented extensive written documentation.

5. Addressing Resistance and Job Security

  • Employee Concerns
    • Some individuals may worry that thorough documentation could threaten their job security.
    • Emphasize that documentation is about knowledge sharing, not replacing roles.

6. Enhancing Documentation with Media

  • Videos

    • Useful for tasks that are easier to demonstrate visually (e.g., complex clicks or steps).
    • Could also serve as quick intros to a project or process.

  • Screen Recordings and Transcripts

    • Tools like SundaySky might provide transcripts for screen recordings, making them searchable.
    • Ensure any text or video elements remain easy to locate and update.

7. Version Control and Updates

  • Need for Versioning

    • Processes evolve, and documentation can quickly become outdated.
    • At a minimum, indicate the last updated date on each document.

  • Regular Reviews

    • A quarterly check of documented processes could help keep them accurate and relevant.

Key Takeaways and Next Steps

  1. Clarify Terminology
    • Decide how to differentiate (or unify) the terms “procedure,” “workflow,” and “process.”
  2. Finalize Tool Choice
    • Continue exploring whether Obsidian, SharePoint, or a combination best meets current and future needs.
  3. Establish Branding and Structure
    • Define a basic template or set of “ingredients” each documentation piece should have.
  4. Address Resistance
    • Communicate the value of documentation for collective knowledge rather than a threat to job security.
  5. Plan for Ongoing Maintenance
    • Implement version tracking and set a schedule (e.g., quarterly) for reviewing and updating documentation.

These notes provide a foundation to guide the next phases of the Data Docs Project, ensuring that all relevant topics—from tool selection to addressing team concerns—are captured and actionable.
