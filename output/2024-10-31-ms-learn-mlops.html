Introduction to DevOps principles for machine learning

Introduction

The increase in machine learning (ML) projects is driven by more accessible data, computational power, and advancements in algorithms. However, a significant challenge in scaling ML projects is the absence of a clear strategy and the existence of organizational silos.

MLOps (Machine Learning Operations) aims to bridge this gap by streamlining the transition from ML proofs of concept to full-scale production. MLOps ensures that ML workflows are robust, reproducible, and scalable by allowing for continuous monitoring, retraining, and redeployment of models. This lifecycle involves training, packaging, validating, deploying, monitoring, and retraining models, facilitated by both data scientists and ML engineers who apply DevOps practices.

DevOps principles such as agile planning, source control, and automation accelerate the ML lifecycle and enhance collaboration. This module introduces DevOps concepts and how tools like Azure DevOps and GitHub can integrate with Azure Machine Learning to optimize ML project management.

Learning objectives:

  • Understand DevOps benefits for ML projects.
  • Learn applicable DevOps principles.
  • Discover integration of Azure DevOps and GitHub with Azure ML.

DevOps for machine learning

The “DevOps for Machine Learning” module discusses integrating DevOps principles into MLOps to improve the scalability, reproducibility, and efficiency of machine learning projects.

Key Concepts:

  1. DevOps and MLOps:

    • DevOps focuses on uniting people, processes, and tools to provide continuous value, emphasizing automation, rapid delivery, and reproducibility.
    • MLOps applies these principles to ML, organizing the ML lifecycle (exploratory data analysis, feature engineering, model training, validation, deployment, monitoring, and retraining).

  2. Three Key Processes in MLOps:

    • ML: Data scientists handle model training, tuning, and feature engineering.
    • DEV: Software development focuses on planning, creating, verifying, and packaging ML models.
    • OPS: Operations handle deployment, monitoring, and infrastructure management.

  3. DevOps Principles:

    • Automation is core, ensuring quick, reproducible deployments through CI/CD (Continuous Integration/Continuous Delivery).
    • Continuous Integration involves code creation, verification, and testing, transforming exploratory code into production-ready scripts.
    • Continuous Delivery automates model deployment, utilizing pre-production staging to ensure reliability before production.
    • Source Control (e.g., Git-based repositories) enables versioning, collaboration, and tracking changes.
    • Agile Planning structures work into sprints, which facilitates quick deployment and iterative improvement.
    • Infrastructure as Code (IaC) standardizes infrastructure configurations, automating environment setup to ensure consistency.

These principles and processes ensure that ML projects remain scalable, collaborative, and adaptable to changes, ultimately enhancing their reliability and alignment with business needs. DevOps tools like Azure DevOps and GitHub support these workflows.

DevOps tools

The “DevOps Tools” module introduces essential tools for implementing DevOps practices, specifically focusing on Azure DevOps, GitHub, and Git, which are valuable for machine learning projects.

Key Tools: Azure DevOps:

A Microsoft platform offering services to streamline DevOps workflows, adaptable across multiple languages and platforms, often used with Python and Linux for ML. Key components include: Azure Boards: Tracks and organizes agile planning with work items, visualizations, and reports. Azure Repos: Stores code in public and private repositories. Azure Pipelines: Supports continuous integration (CI) and continuous delivery (CD) by automating builds and tests. GitHub:

A development platform for open-source projects, combining Git repositories with DevOps tools. Key features: Issues: Manages work items, bugs, and feedback. Repos: Houses public and private code repositories. Actions: Automates workflows to streamline DevOps processes. Git:

A distributed source control system widely used for code management in collaborative projects. Allows trunk-based development with branching, enabling team members to work on separate branches and merge changes back into the main codebase through pull requests. Together, these tools support collaborative development, automation, and efficient code management, which are essential for scaling and maintaining machine learning projects.

Integrate Azure Machine Learning with DevOps tools

This module covers integrating Azure Machine Learning (AML) with Azure DevOps or GitHub, emphasizing the roles and steps required for secure, efficient collaboration.

Key Points:

  1. Roles in Integration:

    • Administrator: Sets up the DevOps environment, manages the tools, and securely connects AML with Azure DevOps or GitHub.
    • End User: Collaborates on project development with restricted access to configuration settings.

  2. Azure DevOps Integration:

    • Setup: Administrators create an organization and projects within Azure DevOps.
    • Service Connection: To connect Azure DevOps to AML, a service principal is created in Microsoft Entra ID, providing a secure identity for AML authentication.
    • Configuration: Within each project, administrators configure the service connection to enable Azure DevOps pipelines to manage the AML workspace securely.

  3. GitHub Integration:

    • Setup: Administrators create a GitHub repository, owned individually or by an organization.
    • Service Principal & Secrets: Using Azure CLI, a service principal with Contributor access to AML is created. The credentials are stored as a GitHub secret (named AZURE_CREDENTIALS) to enable secure connections.
    • Secrets Management: The secret enables GitHub Actions to authenticate and interact with AML workflows seamlessly.

These integrations streamline ML workflows by enabling secure automation for project management, code collaboration, and continuous integration and deployment in AML projects.

Source control for machine learning projects

Introduction

This module discusses using repositories and trunk-based development for managing code in machine learning projects, especially with DevOps tools like Azure DevOps and GitHub.

Key Points:

  1. Repositories:

    • Repositories (repos) store code and enable collaboration. Common choices are Azure Repos in Azure DevOps or GitHub.
    • Repo Scoping:
      • Monorepo: Stores all ML workloads in a single repo, offering easy access for all team members.
      • Multiple Repos: Each ML project has a separate repo, which provides more controlled access but adds management overhead.
    • Folder Structure: Standardized folders within each repo improve navigation, typically including:
      • .cloud for cloud-specific code,
      • .ad/.github for pipeline artifacts,
      • src for ML scripts,
      • docs for documentation,
      • pipelines for AML pipeline definitions,
      • tests for unit/integration tests, and
      • notebooks for experimental Jupyter notebooks.
    • Data Storage: Training data should be stored externally in databases or data lakes, accessible directly by AML.

  2. Trunk-Based Development:

    • A Git workflow where the main branch hosts production code.
    • Developers create branches to modify code, request reviews, and merge approved changes back into the main branch, ensuring only verified updates reach production.

This setup promotes code organization, controlled access, and collaborative development, making it easier for teams to manage ML project workflows.

Work with Azure Repos and GitHub repos

This module explains how to work with Azure Repos and GitHub repositories for code management in machine learning projects, emphasizing trunk-based development and agile planning.

Key Points:

  1. Trunk-Based Development:

    • Best practice for making code changes involves using branches instead of directly modifying the main code. This allows for testing and review before merging updates into production.

  2. Agile Planning:

    • Updates to machine learning models may be required due to issues like data drift. Agile planning tools, such as Azure Boards in Azure DevOps or GitHub Issues, help organize these updates and assign work items.

  3. Working with Azure DevOps:

    • Azure Boards organizes tasks into backlogs, which are assigned to team members and tracked as work items.
    • When assigned a work item, data scientists can create a new branch directly linked to the item. Development is typically done by cloning the branch into an IDE (e.g., Visual Studio Code) for local testing and refinement.

  4. Working with GitHub:

    • GitHub’s Issues tool is used to log tasks, bugs, and feedback within a repository.
    • Team members can assign themselves to issues, create linked branches for each task, and perform local development and testing after cloning the branch to an IDE.

By following these practices, teams ensure organized, tested, and collaborative code updates in ML workflows.

Develop locally with Visual Studio Code

This module details the process of developing code locally for machine learning projects using Visual Studio Code, while maintaining version control with Git-based repositories such as Azure Repos or GitHub.

Key Steps:

  1. Branching for Development:

    • When updating or improving models, data scientists create a separate branch to isolate their work from the production model.

  2. Setting Up Visual Studio Code:

    • Visual Studio Code (VS Code) is a popular IDE for local development. It supports Azure Machine Learning extensions, allowing you to run code on Azure compute resources or your local device, depending on the computational requirements.

  3. Development Process:

    • Clone: Copy the repository (repo) to your local environment using the repo’s URL and the Git clone command in VS Code’s command palette or terminal.
    • Edit and Commit: Make and save code changes, then commit each change with a message for clarity. Small, frequent commits with clear messages improve team understanding.
    • Push: Once changes are committed, push them to update the main repo. You can use the Source Control pane in VS Code for this.

  4. Code Verification:

    • Before pushing, it’s recommended to verify code through linting and unit tests to maintain quality.
    • If other team members have modified the repo, you can pull updates to sync your local version without losing changes.

This workflow helps data scientists maintain control over their contributions and ensures smooth collaboration on ML projects.

Verify your code locally

This module explains verifying code in machine learning projects by running linting and unit testing locally, ensuring quality before changes are deployed or integrated.

Key Steps:

  1. Linting:

    • Purpose: Ensures code follows team standards, checking for syntax and style errors.
    • Using Flake8 with Visual Studio Code:
      • Install Flake8 (pip install flake8).
      • Configure with a .flake8 file to set rules (e.g., line length limits, ignored errors).
      • Errors and warnings are underlined in VS Code, and specific issues can be viewed under the Problems tab.
    • Automated Linting: In Azure Pipelines or GitHub Actions, define CI workflows to automate linting with Flake8.

  2. Unit Testing:

    • Purpose: Verifies functionality by testing individual code components, like data preparation and model training functions.
    • Using Pytest:
      • Write test functions in files (e.g., test_train.py) that check expected outputs using assert statements (e.g., with NumPy assertions for model predictions).
      • Run Tests in VS Code:
        • Configure VS Code for Pytest and run tests from the Testing tab.
    • Automated Testing: In CI pipelines (Azure DevOps or GitHub Actions), run tests using Pytest after installing dependencies.

Verifying code with linting and unit tests ensures quality, prevents errors in production, and aligns with CI/CD workflows by catching issues early.

Introduction

In this module, the focus is on scaling artificial intelligence (AI) by implementing automation through machine learning operations (MLOps). Automation helps transition from experimentation to production, where tasks are organized into pipelines that can be scheduled or event-triggered.

Learning Objectives:

  • Understand how to create and use Azure Machine Learning pipelines.
  • Learn how to automate workflows using Azure Pipelines and GitHub Actions.

Each type of pipeline will be specifically referenced by its full name (e.g., Azure Machine Learning pipelines) to clarify the service in use.

Azure Machine Learning pipelines

This module covers creating and automating machine learning workflows using Azure Machine Learning (AML) pipelines, moving from experimentation to production-ready code.

Key Steps:

  1. Experimentation in AML:

    • AML enables data scientists to register datasets, train models on remote compute resources, track experiment results, and register models for deployment.
    • To transition from experimentation (e.g., Jupyter notebooks) to production, code should be refactored into functions, grouped in Python scripts, and unit-tested.

  2. Creating AML Pipelines:

    • AML pipelines organize ML workflows by grouping tasks into a sequence, which can then be scheduled or triggered.
    • Pipelines are defined in YAML to specify task details:
      • Steps: Defined as Python scripts or reusable components.
      • Inputs/Outputs: Specify data inputs and outputs.
      • Compute Resources: Set up compute environments, often using AML compute clusters.
    • YAML files make pipelines reusable and shareable within the AML workspace.

  3. Pipeline Example and Execution:

    • A sample YAML defines two jobs, transform-job and train-job, each handling data transformation and model training, respectively.
    • Pipelines are executed in AML using CLI commands, enabling integration with automation tools like Azure Pipelines or GitHub Actions for continuous model training and retraining.

This pipeline setup allows teams to automate and scale ML tasks efficiently, preparing models for continuous production deployment.

Azure Pipelines

This module explains how to automate workflows in machine learning projects using Azure Pipelines as part of Azure DevOps. Azure Pipelines can automate tasks such as creating assets, triggering Azure Machine Learning (AML) pipelines, and managing model deployment.

Key Steps: Setting up Azure Pipelines:

Azure Pipelines automate tasks based on triggers (e.g., repo changes). To use Azure Pipelines with AML, a service connection is established between Azure DevOps and the AML workspace. Pipeline Structure:

Pipelines are defined in YAML and include: Trigger: Initiates the pipeline (e.g., repo commit). Stages: Groups of jobs for different environments (dev, test, prod). Jobs: Sets of steps executed on agents (e.g., Microsoft-hosted Ubuntu). Steps: Commands or scripts, such as initiating an AML pipeline using CLI commands. Running AML Pipelines in Azure Pipelines:

A sample YAML file uses az ml job create to run the AML pipeline defined in a file like pipeline-job.yml. Parameters can make YAML files reusable across projects. Configuring and Running the Pipeline:

In Azure DevOps, select the YAML file in the repo to configure and run the pipeline. Once set up, commits to the main branch automatically trigger the pipeline. Job status and error messages can be reviewed in Azure DevOps and the AML workspace. This setup supports seamless, automated workflow execution for machine learning tasks, improving efficiency and scalability in production.

GitHub Actions

This module explains how to use GitHub Actions to automate workflows in machine learning projects hosted on GitHub. GitHub Actions enable tasks such as building, testing, and deploying code, with secure integration into Azure Machine Learning (AML) pipelines.

Key Steps:

  1. Setting Up GitHub Actions:

    • GitHub Actions workflows are defined in YAML files stored in the .github/workflows/ directory of a GitHub repo.
    • Actions are triggered by events (e.g., a code push) and consist of:
      • Jobs: Run on a virtual machine (typically Ubuntu).
      • Steps: Commands or actions to execute, like initiating an AML pipeline.

  2. Example YAML Workflow:

    • A sample YAML workflow file includes:
      • Event Trigger: Runs on a push to the main branch.
      • Steps: Check out the repo, install AML CLI, authenticate with Azure using AZURE_CREDENTIALS, set the working directory, and run the AML pipeline.

  3. Running and Monitoring Workflows:

    • Workflows automatically run when triggered (e.g., by a code push).
    • To monitor workflow results, view the GitHub Actions tab for details on each step’s success or errors.
    • When a workflow triggers an AML pipeline, review the pipeline in the AML workspace to ensure all steps executed correctly.

GitHub Actions simplify continuous integration and deployment for ML projects, helping automate processes and monitor workflows efficiently.

Continuous Deployment for Machine Learning

Introduction

MLOps facilitates the transition of machine learning projects from experimentation to scalable production.

Key Concepts:

  • Experimentation: Models are tested and refined in a flexible development environment.
  • Production: Models are deployed in a scalable, robust environment designed for long-term use.

Objective:

  • Automate the transition from development to production with continuous delivery.

Learning Objectives for the Module:

  1. Set up development and production environments.
  2. Control deployments using approval gates to ensure readiness at each stage.

Set up environments for development and production

This module explains setting up and managing development and production environments within DevOps to support continuous delivery in machine learning workflows.

Key Concepts:

  1. Using Environments in MLOps:

    • Dev (Development): Experiment with model training and testing.
    • Pre-Prod/Staging: Deploy and test the selected model.
    • Prod (Production): Release the model for end-user consumption.
    • Separate environments aid in access control, often using role-based access control (RBAC) for resource management.

  2. Organizing Environments in Azure Machine Learning (AML):

    • Each environment can have a dedicated AML workspace for better access control and resource isolation.
    • Alternatively, a single workspace can be used for all stages, though this increases shared access risks.

  3. Setting up Environments in Azure DevOps:

    • Create environments in Azure DevOps Pipelines and specify the target environment (e.g., dev, prod) within the pipeline YAML.
    • Define the service connection in the YAML to link to the correct AML workspace.

  4. Using Environments in GitHub Actions:

    • Define environments within GitHub repository settings.
    • Use environment secrets to securely connect each environment to its designated AML workspace.
    • Specify the target environment in the GitHub Actions YAML file to control workflow deployment stages.

These environment setups help streamline model deployment stages, enhance security, and improve scalability for ML projects, facilitating a structured pathway from development to production.

Control deployments with approval gates

This module explains how to manage machine learning model deployments by using approval gates within separate environments (development, staging, and production).

Key Concepts:

  1. Benefits of Approval Gates:

    • Improve resource access control and deployment oversight.
    • Enable manual approval checks between environment stages, ensuring models are reviewed before progressing from development to staging and production.

  2. Setting Up Approval Checks:

    • Azure Pipelines:
      • Navigate to the environment, select Approvals and checks, and assign approvers.
      • Approvers are notified to permit the pipeline to run.
    • GitHub Actions:
      • In the environment settings, enable required reviewers and assign GitHub users as approvers.
      • Approvers review and approve workflows before deployment proceeds.

Approval gates provide control over continuous deployment, ensuring that model changes are vetted at each stage before production.

Example

Here’s a comprehensive guide for a small team of two managers, two data scientists, and three data analysts on how to approach the ML Life Cycle using Azure Machine Learning Studio and Azure DevOps.

Team Structure:

  • Managers: Oversee business goals, approve resources, and align model development with business objectives.
  • Data Scientists: Lead model development, handle algorithm selection, data preparation, model tuning, and evaluate model performance.
  • Data Analysts: Perform data exploration, feature engineering, and manage data insights to guide modeling.

ML Life Cycle Stages with Azure ML Studio and Azure DevOps

1. Discovery Stage

Goal: Define the business use case, establish evaluation metrics, and validate data readiness.

Step-by-Step:

  1. Define Business Objectives (Managers & Data Scientists):

    • Managers and data scientists work together to outline the business goals (e.g., customer retention).
    • In Azure DevOps, use Azure Boards to document business goals and establish key performance indicators (KPIs).

  2. Data Exploration and Analysis (Data Analysts):

    • Analysts assess data quality and distribution using Azure Machine Learning Studio to load and explore datasets.
    • Data profiling can be done with Azure ML Studio’s data exploration tools, using visualizations like histograms and scatter plots to identify patterns or issues (e.g., missing values, outliers).

  3. Preliminary Feasibility Study (Data Scientists):

    • Run initial models or simple algorithms to see if a viable signal exists within the data.
    • Use Jupyter notebooks in Azure ML Studio to test algorithms quickly and store results for review.
    • Document results in Azure DevOps as a reference for future development.

Outputs:

  • Defined business goals and KPIs.
  • Initial data insights and feasibility report.

2. Development Stage

Goal: Build data pipelines, engineer features, and develop a robust model that aligns with business KPIs.

Step-by-Step:

  1. Data Pipelines and Feature Engineering (Data Analysts & Data Scientists):

    • Analysts and data scientists collaboratively use Azure ML Studio to set up pipelines that automate data preprocessing.
    • Azure DevOps Pipelines can be integrated to trigger these pipelines automatically upon data updates, ensuring data quality checks are met.
    • Use feature engineering to transform raw data into model-ready formats (e.g., one-hot encoding, handling missing values).

  2. Model Development (Data Scientists):

    • Develop multiple models (e.g., linear regression, decision trees) and tune parameters within Azure ML Studio.
    • Azure Machine Learning Pipelines allows defining repeatable workflows for model training, which can be saved and reused in future iterations.
    • Set up unit tests in Azure DevOps to verify model scripts, ensuring consistent data transformations and model predictions.

  3. Evaluation and Refinement (Data Scientists):

    • Compare model performance against evaluation metrics established in the discovery stage.
    • Track experiments in Azure ML Studio to record performance metrics, model configurations, and parameter tuning efforts.
    • Use Azure DevOps for version control, creating a Git repository to save code, model versions, and data preprocessing scripts.

Outputs:

  • Documented data pipelines and feature engineering steps.
  • Trained and tested model with recorded results and metrics.

3. Deployment Stage

Goal: Deploy the model to production, automate workflows, and establish monitoring.

Step-by-Step:

  1. Planning for Deployment (Managers & Data Scientists):

    • Managers approve resources for deployment, deciding on real-time vs. batch inference requirements based on business needs.
    • Use approval gates in Azure DevOps to ensure all deployments meet quality and performance standards before moving to production.

  2. Model Operationalization with MLOps (Data Scientists):

    • Set up a CI/CD pipeline in Azure DevOps to automate model deployment.
    • Define environment configurations in Azure ML Studio, setting up distinct environments (dev, pre-prod, and prod) with appropriate access control.
    • Use Azure DevOps Pipelines to trigger model deployment workflows to pre-prod for testing and final approval before production.

  3. Monitoring and Maintenance (Data Analysts):

    • Set up Azure Machine Learning monitoring to track model performance over time, checking for data drift and model staleness.
    • Azure ML Studio’s monitoring tools allow analysts to track feature distributions and set alerts if data characteristics deviate from the training data.
    • Regularly update models and retrain as needed, using Azure DevOps to trigger retraining pipelines based on monitored metrics.

  4. Continuous Improvement (Managers):

    • Review model impact on business objectives, analyzing how it aligns with KPIs.
    • Use Azure DevOps to create a backlog of any new requirements or improvements, fostering an iterative improvement loop that cycles back to the discovery stage as necessary.

Outputs:

  • Production-ready model with a CI/CD pipeline for retraining and updates.
  • Monitoring systems to track data drift and model accuracy over time.

Summary Workflow Example:

Initial Setup:

  • Define business objectives and KPIs in Azure Boards.
  • Data ingestion and exploration using Azure ML Studio.

Development and Testing:

  • Data analysts build preprocessing pipelines and feature engineering steps.
  • Data scientists create model experiments and track metrics in Azure ML Studio.
  • Use Azure DevOps for version control, model validation, and automated testing.

Deployment:

  • Approval gates ensure all deployment standards are met.
  • Deploy model to production with Azure Pipelines and set up monitoring.

Maintenance and Retraining:

  • Continuously monitor model performance.
  • Trigger retraining workflows based on drift detection and feedback, ensuring the model stays relevant and impactful.
